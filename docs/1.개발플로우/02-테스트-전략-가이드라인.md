# 테스트 전략 가이드라인

본 문서는 AI 주도 개발 환경에서의 테스트 전략과 Frontend Store TDD의 선택적 사용 기준을 정의합니다.

---

## 핵심 원칙

### Testing Trophy (최신 트렌드)

```
        ╱╲  E2E (핵심 흐름만)
       ╱──╲
      ╱────╲  Integration (주력)
     ╱──────╲
    ╱────────╲  Unit (필요한 곳만)
   ╱──────────╲
  ╔════════════╗  Static Analysis (TypeScript, ESLint)
```

> "Write tests. Not too many. Mostly integration." - Guillermo Rauch (Vercel CEO)

### 테스트 레이어별 역할

| 레이어 | 테스트 방식 | 담당 |
|--------|------------|------|
| **Backend API** | TDD 완전 검증 (필수) | `api/.../test/` |
| **Frontend Store** | TypeScript 타입 체크 | `vue-tsc --noEmit` |
| **Frontend UI** | E2E 핵심 플로우 (선택) | Playwright/Cypress |
| **Store 단위 테스트** | 복잡한 클라이언트 로직 있을 때만 | 선택적 |

## Backend TDD vs Frontend Store TDD

### Backend TDD (필수)

```
┌─────────────────────────────────────────────────────────────────┐
│  Backend TDD의 역할                                              │
│                                                                 │
│  ✅ 비즈니스 로직 완전 검증                                      │
│  ✅ DB 쿼리 정확성 검증                                          │
│  ✅ API 응답 구조 검증                                           │
│  ✅ 에러 처리 검증                                               │
│                                                                 │
│  → 모든 CRUD 모듈에서 필수                                       │
└─────────────────────────────────────────────────────────────────┘
```

### Frontend Store TDD (선택적)

```
┌─────────────────────────────────────────────────────────────────┐
│  Store TDD가 불필요한 경우 (대부분)                              │
│                                                                 │
│  - Store가 단순 API Wrapper인 경우                              │
│  - 비즈니스 로직이 Backend에 있는 경우                          │
│  - Store 함수가 useApi().get/post/put/delete만 호출하는 경우    │
│                                                                 │
│  예시 (테스트 불필요):                                          │
│  async paging() {                                               │
│    const result = await useApi().get('/api/...', params);       │
│    this.listData = result.data;  // 단순 할당                   │
│  }                                                              │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Store TDD가 필요한 경우 (드묾)                                  │
│                                                                 │
│  - 복잡한 클라이언트 사이드 계산 로직                           │
│  - 여러 API 응답을 조합/변환하는 로직                           │
│  - 상태 변환 로직 (State Machine)                               │
│  - 캐싱/메모이제이션 로직                                       │
│                                                                 │
│  예시 (테스트 필요):                                            │
│  computed: {                                                    │
│    totalUnpaidAmount(): number {                                │
│      return this.listData                                       │
│        .filter(item => item.payState === 'REQ')                 │
│        .reduce((sum, item) => sum + this.calcLateFee(item), 0); │
│    },                                                           │
│    calcLateFee(item: Payment): number {                         │
│      const days = differenceInDays(new Date(), item.dueDate);   │
│      return item.amount * 0.05 * Math.min(days, 30);            │
│    }                                                            │
│  }                                                              │
└─────────────────────────────────────────────────────────────────┘
```

## AI 주도 개발 관점

### Store TDD의 문제점

```
AI가 Store TDD를 이해하려면 참조해야 하는 파일:

1. front/.../test/[모듈명].test.ts    (Frontend)
2. front/.../store/[모듈명].store.ts  (Frontend)
3. api/.../service/[모듈명]-tdd.service.ts  (Backend)
4. api/.../dao/[모듈명]-tdd.dao.ts    (Backend)
5. api/.../service/[모듈명].service.ts (Backend)

→ 5개 파일 참조 필요 = "독립성" 원칙 위반
```

### 권장 전략

```yaml
독립성 확보:
  - Backend TDD: 해당 모듈 폴더만 참조하면 완전 이해 가능
  - Frontend: TypeScript 정적 분석으로 대부분 커버

효율성:
  - 동일 API를 두 번 테스트하지 않음 (Backend에서 한 번만)
  - 테스트 유지보수 비용 절감
```

## 실무 적용 가이드

### 1. 새 모듈 개발 시

```bash
# 1. Backend TDD 먼저 완료 (필수)
cd api && bun test src/modules/[모듈명]/test/

# 2. Frontend Store 생성 (TDD 없이)
/gen-store [모듈명]

# 3. 타입 체크로 검증
cd front && npx vue-tsc --noEmit
```

### 2. Store TDD가 필요한 경우 판단 기준

| 질문 | 예 → TDD 필요 | 아니오 → TDD 불필요 |
|------|--------------|-------------------|
| Store에 복잡한 계산 로직이 있는가? | ✅ | ❌ |
| 여러 API 응답을 조합하는가? | ✅ | ❌ |
| 상태 변환 로직이 복잡한가? | ✅ | ❌ |
| Store가 단순 API Wrapper인가? | ❌ | ✅ |

### 3. 기존 Store TDD 처리

```yaml
기존 Store TDD 파일이 있는 경우:
  - 삭제 권장 (Backend TDD로 충분)
  - 유지해도 무방 (실행 안 해도 됨)

신규 모듈:
  - Store TDD 생성 안 함 (기본)
  - 복잡한 로직 있으면 수동 추가
```

## Controller TDD API

### API 유지 이유

```
Controller의 TDD API (/tdd/init, /tdd/cleanup)는 유지합니다:

1. Backend TDD에서 사용
2. 향후 Frontend에서 필요시 사용 가능
3. 수동 테스트 데이터 생성에 활용 가능
```

### 사용 시나리오

| 시나리오 | TDD API 사용 여부 |
|---------|------------------|
| Backend TDD | ✅ TddService 직접 호출 |
| Frontend Store TDD (복잡한 로직) | ✅ Controller API 호출 |
| 개발 중 수동 테스트 데이터 생성 | ✅ API 호출 가능 |
| 일반 Frontend 개발 | ❌ 사용 안 함 |

## Pinia 공식 권장사항

### Store 테스트보다 컴포넌트 테스트

Pinia 공식 문서에서는 `createTestingPinia`를 사용해 Store를 Mock 처리하고,
Store를 사용하는 **컴포넌트**를 테스트하는 것을 권장합니다.

```typescript
// Pinia 권장: Store Mock으로 컴포넌트 테스트
import { mount } from '@vue/test-utils'
import { createTestingPinia } from '@pinia/testing'

const wrapper = mount(MyComponent, {
  global: {
    plugins: [createTestingPinia()],
  },
})

const store = useSomeStore()  // Mock Store
store.someAction()
expect(store.someAction).toHaveBeenCalledTimes(1)
```
