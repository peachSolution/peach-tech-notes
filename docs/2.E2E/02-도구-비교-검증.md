# 브라우저 자동화 도구 비교 검증

Claude Code에서 사용 가능한 브라우저 자동화 도구를 **실측 검증**한 결과.

---

## 운용 원칙

### Chrome Beta로 분리 운용

기존 Chrome은 평소대로 사용하고, E2E·자동화 전용 브라우저를 Chrome Beta로 별도 분리하여 사용 편의를 도모한다.

### 사람이 로그인, AI가 이어받는다

모든 브라우저 자동화의 시작은 **사람이 직접 로그인**하는 것이다.
AI는 로그인된 브라우저를 이어받아 내용 확인·제어·데이터 검증을 수행한다.
브라우저는 닫지 않고 유지하여, 여러 세션에 걸쳐 AI가 이어서 사용한다.

### 두 가지 핵심 용도

1. **AI 브라우저 탐색** — 개발한 프로그램·API·외부 사이트를 AI가 탐색하고 내용을 확인하거나 데이터를 점검한다 → **playwright-cli** (Chrome Beta, `--persistent`, close 안 함)
2. **E2E 반자동 테스트** — 스크립트로 만들어두고 headed 모드로 실행하여 진행 과정을 눈으로 보면서, 처리된 데이터를 이어서 확인한다 → **Playwright JS API** (Chrome Beta CDP, headed)

---

## 왜 이 방법론이 효과적인가

### 기존 E2E 테스트의 문제

| 기존 방식 (headless + CI) | 문제점 |
|--------------------------|--------|
| headless 실행 → 로그만 확인 | 뭐가 잘못됐는지 로그만으로 파악 어려움 |
| 실패 시 스크린샷 뒤져보기 | 실패한 시점의 화면 상태를 재현하기 힘듦 |
| 테스트 완료 후 브라우저 종료 | 테스트가 만든 데이터를 확인하려면 별도 접속 필요 |
| 매번 새 브라우저 실행 | 로그인 반복, 맥락 단절 |

### AI 보조 반자동 방식이 해결하는 것

| headed + 브라우저 유지 + AI 이어받기 | 효과 |
|-----------------------------------|------|
| 실행 과정을 눈으로 보면서 진행 | 이상한 동작을 **즉시 포착** |
| 테스트 완료 후 브라우저가 살아있음 | 처리된 데이터를 **그 자리에서 바로 확인** |
| AI가 이어받아 추가 검증 | 수동 클릭 없이 **AI에게 "이거 확인해줘" 지시** |
| 로그인 세션 영구 유지 | 반복 로그인 시간 **절약**, 맥락 **유지** |
| 진행 과정 → 데이터 확인이 하나의 흐름 | headless 후 별도 확인하는 **이중 작업 제거** |

### 테스트 방법론 분류

| 방법론 | 설명 | 적합 상황 |
|--------|------|----------|
| 완전 자동 (headless CI) | 사람 개입 없음, 결과만 확인 | 회귀 테스트, 대량 반복 |
| 완전 수동 | 사람이 직접 클릭 | 탐색적 테스트, UX 검증 |
| **AI 보조 반자동 (채택)** | **사람이 로그인 → AI가 실행 → 사람이 결과 확인** | **개발 중 검증, 데이터 확인, API 탐색** |

> **AI 보조 반자동 방식**은 "개발한 프로그램이 의도대로 동작하는지 빠르게 확인"하는 데 최적이다.
> headless CI는 "이미 검증된 시나리오를 대량 반복"하는 단계에서 사용한다.

## 일회성 탐색에 가장 효과적인 도구

"AI가 로그인된 사이트에서 이것저것 확인해주는" 일회성 탐색 시나리오에서 실측 비교한 결과:

| 도구 | 일회성 탐색 적합도 | 이유 |
|------|:----------------:|------|
| **playwright-cli** | ★★★ 최적 | CDP 연결로 기존 Chrome Beta 재사용, ref 기반 안정적, iframe eval 가능, 여러 세션에 걸쳐 이어서 사용 |
| browser-use CLI | ★★ | CDP로 기존 브라우저에 즉시 연결 가능, eval 지원, 단 매 명령마다 `--cdp-url` 필요 |
| agent-browser CLI | ★★ | snapshot ref 안정적이나 iframe 접근 불가, 단순 페이지에서만 유효 |
| claude-in-chrome MCP | ★ | 스크린샷 기반 시각 판단은 강력하나 느리고 토큰 소비 높음, 도구 로드에 ToolSearch 매번 필요 |

**playwright-cli를 기본 탐색 도구로 채택한 이유:**
- **CDP 연결 지원** → Playwright JS API와 **동일한 Chrome Beta 1개**로 통합 운용
- 사람이 한 번 로그인하면 **세션이 영구 유지**되고 브라우저를 닫지 않아도 됨
- AI가 `snapshot → click ref → eval` 패턴으로 **스크립트 없이** 무엇이든 확인 가능
- Chrome Beta를 사용하므로 **기존 Chrome과 분리** 운용
- Microsoft 공식 도구로 **장기 지원** 기대

## CDP 프로토콜의 탭 포커스 한계 (검증 완료)

CDP 원격 디버깅 프로토콜은 **사용자가 Chrome에서 현재 보고 있는 탭(포커스 탭)을 알 수 없다.**

| CDP API | 반환값 | 포커스 정보 |
|---------|--------|:-----------:|
| `GET /json` | 탭 목록, URL, title | ❌ |
| `Target.getTargets` | attached 여부 | ❌ (모든 탭이 true) |
| `Browser.getWindowForTarget` | 윈도우 위치/크기 | ❌ |

- playwright-cli의 `(current)` 표시 → Chrome UI 포커스가 **아니라** playwright-cli가 마지막으로 조작한 page 객체
- `### Page` 섹션의 URL/Title → 내부 page 객체 고정값이므로 실제 탭과 **불일치**
- 탭 포커스 정보는 Chrome Extension API(`chrome.tabs.query({active: true})`)로만 접근 가능

**대응 방법:**
- AI 탐색 시 → `tab-list`로 탭 목록을 사용자에게 보여주고, 사용자가 탭 번호를 지정하면 `tab-select`로 전환
- 시나리오 실행 시 → `connect.js`에 탭 인덱스를 전달하거나, URL 패턴으로 자동 탐지

### playwright-cli eval 제한 (검증 완료)

`pwc eval`은 **단순 표현식만 지원**한다. IIFE `(function(){...})()` 패턴은 직렬화 오류가 발생한다.

```bash
# ❌ IIFE — 오류 발생
pwc eval "(function(){var d=document.querySelector('iframe').contentDocument; return d.title})()"
# Error: page._evaluateFunction: Passed function is not well-serializable!

# ✅ 단순 표현식 체이닝
pwc eval "document.querySelector('iframe').contentDocument.title"
```

여러 동작이 필요하면 각각 별도 eval로 나눠서 실행한다.

## 도구 5종 + Playwright 4종 구분

### 브라우저 자동화 도구


| 도구                       | 방식                              | CDP 연결             | 세션 유지                          | AI 개입        |
| ------------------------ | ------------------------------- | ------------------ | ------------------------------ | ------------ |
| **Playwright JS API**    | Node.js 스크립트 (`connectOverCDP`) | ✅                  | CDP 세션                         | 없음 (Bash 실행) |
| **playwright-cli**       | 셸 명령어 (`snapshot → click ref`)  | ✅ (설정 파일/환경변수)    | CDP 세션 또는 `--persistent` 프로필   | 매 단계         |
| **browser-use CLI**      | 셸 명령어 (`--cdp-url`)             | ✅                  | CDP 세션                         | 매 단계         |
| **agent-browser CLI**    | 셸 명령어 (`AGENT_BROWSER_CDP=`)    | ✅                  | CDP 세션                         | 매 단계         |
| **claude-in-chrome MCP** | Chrome 확장 프로그램 경유               | ❌ (별도 Chrome)      | Chrome 확장                      | 매 단계 판단      |


### Playwright 4종은 각각 별개의 도구


| 구분          | Playwright MCP  | Playwright CLI (본체) | playwright-cli (AI용)                                           | Playwright JS API |
| ----------- | --------------- | ------------------- | -------------------------------------------------------------- | ----------------- |
| **패키지**     | @playwright/mcp | npx playwright (내장) | [@playwright/cli](https://github.com/microsoft/playwright-cli) | playwright npm    |
| **실행**      | MCP 서버 (장기)     | 셸 명령 (codegen 등)    | **셸 명령 (snapshot, click, fill)**                               | Node.js 스크립트      |
| **CDP 연결**  | ❌               | ❌                   | **✅ (설정 파일 `cdpEndpoint`)**                                     | ✅ connectOverCDP  |
| **브라우저 조작** | ✅ MCP 도구        | ❌ (녹화/캡처만)          | **✅ ref 기반 직접 조작**                                             | ✅ 프로그래밍 API       |
| **토큰 소비**   | ~114,000        | ~27,000             | **낮음 (스냅샷 텍스트)**                                               | **0**             |
| **핵심 용도**   | AI 샌드박스 환경      | codegen 녹화, 스크린샷    | **AI 에이전트 브라우저 조작**                                            | **E2E 반복 자동화**    |


> **주의**: `npx playwright`(본체)와 `playwright-cli`(@playwright/cli)는 **완전히 다른 패키지**다.
> 본체는 테스트 프레임워크 CLI, playwright-cli는 AI 에이전트 전용 브라우저 조작 도구.

## 용도별 추천


| 용도 | 추천 도구 | 이유 |
|------|----------|------|
| **AI 브라우저 탐색** (사이트 확인, 데이터 점검, API 체크) | **playwright-cli** | CDP 연결로 동일 Chrome Beta 사용, ref 기반, iframe eval 가능 |
| **E2E 반자동 테스트** (스크립트 실행 + 눈으로 확인) | **Playwright JS API** | 가장 빠름, headed CDP, iframe 완벽, 토큰 0 |
