# Page Agent 조사 정리

> Alibaba의 `page-agent` 저장소를 기준으로 정리한 브라우저 내 자연어 GUI 에이전트 분석 메모

---

## 한 줄 정의

`Page Agent`는 **웹페이지 안에 직접 들어가서**, 자연어로 버튼 클릭·입력·선택 같은 UI 조작을 수행하게 만드는 **클라이언트사이드 GUI 에이전트**다.

중요한 기준:
- 브라우저 전체를 돌리는 범용 자동화 엔진이라기보다
- **내 웹서비스 안에 붙이는 AI 코파일럿/조작 레이어**에 가깝다

---

## 이 저장소가 하는 역할

공식 README 기준 핵심 역할은 다음과 같다.

1. 웹페이지 내부에서 DOM을 읽고
2. 자연어 지시를 해석해
3. 실제 UI 동작으로 바꾸는 것

즉, 사용자가
- "로그인 버튼 눌러"
- "이 폼 채워"
- "다음 단계로 넘어가"

같이 말하면, 페이지 안에서 그 작업을 수행하게 만드는 라이브러리다.

---

## 어떤 문제를 풀려고 하나

기존 브라우저 자동화는 보통 아래 둘 중 하나였다.

- 테스트 프레임워크처럼 브라우저 밖에서 코드를 실행하는 방식
- RPA처럼 브라우저 또는 화면 전체를 자동화하는 방식

`Page Agent`는 그 중간이 아니라 **방향 자체가 다르다**.

해결하려는 문제:
- 웹서비스 안에 AI 코파일럿을 빠르게 붙이고 싶다
- 복잡한 폼 입력이나 반복 UI 조작을 자연어 한 문장으로 줄이고 싶다
- 접근성 보조처럼 자연어/음성으로 웹앱을 조작하게 만들고 싶다

즉:
- `Page Agent`는 **제품 내부 기능**을 만드는 도구다
- 운영 배치 자동화나 범용 웹 순회 엔진이 아니다

---

## 대표 사용 시나리오

공식 문서 기준으로 정리하면 아래 시나리오에 가장 잘 맞는다.

### 1. SaaS AI 코파일럿

내 서비스 화면 안에 AI 비서를 붙여서,
사용자가 자연어로 기능을 실행하게 만든다.

예:
- 관리자 페이지에서 고객 검색
- 설정 화면에서 옵션 변경
- 대시보드 필터 자동 설정

### 2. 복잡한 폼 자동 입력

ERP, CRM, 백오피스처럼 클릭 수가 많은 화면에서:
- 여러 입력 필드 채우기
- 셀렉트 박스 선택
- 단계별 저장 버튼 클릭

을 한 문장으로 줄이는 용도다.

### 3. 자연어 접근성 보조

기존 웹앱을 자연어 명령이나 음성 명령으로 조작하게 해,
접근성을 높이는 방향에 맞는다.

### 4. 멀티페이지 작업 확장

기본은 현재 페이지 중심이지만,
Chrome 확장을 함께 쓰면 탭 간 작업이나 브라우저 레벨 작업까지 확장할 수 있다.

### 5. MCP를 통한 외부 제어

MCP 서버를 붙이면 Claude Desktop 같은 외부 에이전트가
브라우저 안의 `Page Agent` 확장 계층을 통해 작업을 실행할 수 있다.

---

## 어떻게 구성되어 있나

2026-04-15 기준 저장소는 모노레포이며, 실제 패키지 구성은 아래와 같다.

- `packages/page-agent`
- `packages/core`
- `packages/page-controller`
- `packages/llms`
- `packages/ui`
- `packages/extension`
- `packages/mcp`
- `packages/website`

실무적으로 이해하면 아래 4계층으로 보면 된다.

| 계층 | 역할 |
|------|------|
| `page-agent` / `core` | 페이지 안에서 동작하는 핵심 에이전트 로직 |
| `page-controller` / `ui` / `llms` | 실제 제어, UI 패널, LLM 연동 보조 계층 |
| `extension` | 멀티페이지, 브라우저 레벨 작업을 위한 확장 계층 |
| `mcp` | 외부 AI 클라이언트가 확장을 통해 브라우저를 제어하게 하는 브리지 |

정리하면:
- 기본형: 웹페이지 안에서 동작
- 확장형: Chrome extension으로 범위 확대
- 외부연동형: MCP로 다른 에이전트와 연결

---

## Page Agent vs Playwright vs browser-use

헷갈리기 쉬운 세 도구를 역할 기준으로 비교하면 아래와 같다.

| 도구 | 실행 위치 | 제어 방식 | 가장 잘 맞는 용도 | 잘 안 맞는 용도 |
|------|----------|----------|------------------|----------------|
| `Page Agent` | 웹페이지 안 | DOM 기반 자연어 UI 조작 | 제품 내부 AI 코파일럿, 폼 입력 보조, 자연어 UI | 범용 RPA, 서버 배치 자동화 |
| `Playwright` | 브라우저 밖 | 코드 기반 정밀 제어 | E2E 테스트, 회귀 테스트, 재현성 높은 자동화 | 제품 안에 넣는 자연어 조작 계층 |
| `browser-use` | 브라우저 밖 | AI 에이전트 기반 브라우저 작업 | 웹 작업 자동화, 에이전트 워크플로우, 클라우드 실행 | 제품 내부 UI 컴포넌트처럼 직접 붙이는 용도 |

핵심 차이:
- `Page Agent`는 **내 서비스 안에 넣는 도구**
- `Playwright`는 **개발/테스트 자동화 도구**
- `browser-use`는 **AI가 웹을 돌아다니며 작업하는 도구**

`Page Agent` README도 스스로를
- `client-side web enhancement`
로 규정하고,
- `server-side automation`은 아니라고 선을 긋는다.

## 이 도구를 과장하면 안 되는 이유

이 저장소를 소개할 때 가장 흔한 오해는:
- "이거면 브라우저 자동화 다 되는 거 아닌가?"

하지만 리스크가 있다.

### 과대해석하면 안 되는 지점

- `한 줄 스크립트`로 붙는다고 해서 범용 자율 에이전트가 되는 것은 아니다
- `MCP Server`는 공식 문서 기준 `Beta`다
- `Chrome extension`은 기본 필수 기능이 아니라 `optional` 확장이다
- 어떤 웹사이트에서든 100% 안정적으로 작동하는 범용 RPA로 기대하면 안 된다

### 공식 문서상 한계

- 기본은 **단일 페이지 중심**
- 멀티페이지는 **확장 프로그램 필요**
- DOM 텍스트 기반이라 hover, drag-and-drop, canvas 같은 상호작용은 제한이 있다
- 데모 CDN과 테스트 LLM은 운영용이 아니라 **기술 평가용**
- IIFE 빌드에서 `.env` 값을 그대로 쓰면 API 키 인라인 노출 위험이 있다

---

## 언제 잘 맞는가

아래 조건이면 `Page Agent`가 맞다.

- 우리가 제어 가능한 웹앱 UI 안에 AI 비서를 넣고 싶다
- 현재 페이지 중심으로 동작해도 충분하다
- 복잡한 폼 입력이나 반복 클릭을 자연어로 줄이고 싶다
- 음성/자연어 접근성 보조 기능이 필요하다

즉:
- **제품 내부 UX 개선 도구**로 보면 맞다

---

## 언제 쓰면 안 되는가

아래 조건이면 다른 도구를 먼저 봐야 한다.

- 여러 외부 사이트를 장시간 순회해야 한다
- 브라우저 전체 자동화가 핵심이다
- 테스트 재현성과 회귀 검증이 최우선이다
- 프록시, 스텔스, 대규모 병렬 실행 같은 운영형 브라우저 자동화가 중요하다

이 경우:
- 테스트면 `Playwright`
- AI 기반 외부 웹 자동화면 `browser-use`
쪽이 더 맞다.

---

## 도입 전에 확인할 질문

실무에서는 아래 3개를 먼저 물으면 된다.

1. 우리 요구가 **페이지 내부 보조 기능**인가, **브라우저 전체 자동화**인가?
2. 멀티페이지/멀티탭이 필수인가, 선택인가?
3. 운영 환경에서 직접 LLM 키, 프라이버시, 레이트리밋, 오리진 제한을 감당할 수 있는가?

이 질문에 답하면 도입 여부가 거의 정리된다.

---

## 지금 기준 결론

맞습니다.

`Page Agent`는
- `Playwright`의 대체재도 아니고
- 범용 RPA 엔진도 아니며
- `browser-use`와 동일한 위치의 도구도 아니다.

정확한 포지션은:

**"웹서비스 안에 붙는 자연어 기반 GUI 에이전트"**

한 줄 비유:

**웹사이트 안에 상주하면서 사람 말로 버튼과 폼을 대신 다뤄주는 안내원**

이 기준으로 이해하면 맞고,
브라우저 전체 자동화 플랫폼으로 기대하면 리스크가 있다.
