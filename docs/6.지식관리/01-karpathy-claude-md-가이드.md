# Karpathy 영감 CLAUDE.md 가이드

> 대상 독자: Claude Code, Codex, 유사 코딩 에이전트를 실무에 도입하려는 개발자
> 목표: 이 문서 하나만 읽고 여러 프로젝트에 바로 적용할 수 있게 정리한다.

---

## 왜 이 방식이 필요한가

코딩 에이전트는 반복적으로 비슷한 실수를 한다.

- 물어보지 않고 가정한다
- 요청보다 크게 만든다
- 관련 없는 코드까지 고친다
- 무엇을 성공으로 볼지 정하지 않고 진행한다

Karpathy 영감 `CLAUDE.md` 접근은 이 문제를 코드가 아니라 `행동 규칙`으로 줄이려는 시도다.

핵심은 Markdown 파일 자체가 아니다.  
핵심은 에이전트가 매번 흔들리는 지점을 짧고 일관된 규칙으로 고정하는 데 있다.

---

## 핵심 결론

### 1. `CLAUDE.md`는 프로젝트 설명서가 아니라 행동 제어층이다

이 파일에는 저장소 전체 설명을 길게 넣기보다, 에이전트가 어떻게 행동해야 하는지를 짧게 넣는 편이 낫다.

예:

- 모호하면 추측하지 말고 질문한다
- 요청 밖 기능을 추가하지 않는다
- 변경한 모든 줄은 요청과 직접 연결되어야 한다
- 작업 전 성공 기준을 정하고, 작업 후 그 기준으로 검증한다

### 2. `CLAUDE.md` 하나만으로 끝나지 않는다

실무에서는 보통 아래처럼 역할을 나눠야 유지가 된다.

- `CLAUDE.md`: 공통 행동 원칙
- `AGENTS.md`: 저장소 구조, 운영 규칙, 파일 배치 기준
- `.claude/rules/`: 경로별, 주제별 세부 규칙
- `CLAUDE.local.md`: 개인 로컬 메모와 개인 설정

### 3. `AGENTS.md`를 이미 쓰고 있다면 import 하는 편이 안전하다

같은 규칙을 여러 곳에 복사하면 금방 어긋난다.  
저장소 사실 정보는 `AGENTS.md`에 두고, `CLAUDE.md`에서 참조하는 방식이 실무적으로 가장 덜 깨진다.

예시:

```md
@AGENTS.md

## 작업 원칙
- 모호하면 추측하지 말고 질문한다.
- 단순 작업은 과잉 질문하지 않는다.
- 요청 범위 밖 기능, 리팩터링, 의존성 추가를 하지 않는다.
- 변경한 모든 줄은 요청과 직접 연결되어야 한다.
- 작업 전 성공 기준을 짧게 정리하고, 작업 후 그 기준으로 확인한다.
```

### 4. 모든 작업에 같은 강도로 적용하면 느려진다

이 방식은 중간 이상 복잡도의 작업에서 가장 효과적이다.

- 작은 작업: 가볍게
- 중간 작업: 적극 적용
- 고위험 작업: 강하게 적용

작은 작업까지 무겁게 다루면 질문만 늘고 속도는 떨어진다.

### 5. 문서는 짧고 구체적일수록 잘 작동한다

`CLAUDE.md`에 배경 설명, 긴 절차, 경로별 예외, 개인 취향까지 한꺼번에 넣으면 준수율이 떨어진다.

정리 원칙:

- 공통 행동 규칙만 남긴다
- 긴 절차는 별도 문서로 뺀다
- 경로별 규칙은 `.claude/rules/`로 보낸다
- 개인 설정은 `CLAUDE.local.md`로 뺀다

## Karpathy 4원칙의 실무 해석

### 1. Think Before Coding

뜻:

- 가정을 숨기지 않는다
- 여러 해석이 가능하면 조용히 하나를 고르지 않는다
- 더 단순한 길이 있으면 먼저 말한다
- 진짜로 모호하면 질문한다

실무 효과:

- 잘못된 전제로 멀리 가는 문제를 줄인다
- "왜 이렇게 구현했지?"라는 결과를 줄인다

주의:

- 모든 작업에서 질문을 늘리라는 뜻은 아니다
- 결과를 바꾸는 모호함만 질문한다

### 2. Simplicity First

뜻:

- 요청하지 않은 기능 추가 금지
- 단일 사용 코드에 과한 추상화 금지
- 미래를 위한 유연성 추가 금지
- 더 짧게 가능하면 줄인다

실무 효과:

- 첫 구현부터 diff가 작아진다
- 리뷰에서 과설계 지적이 줄어든다

주의:

- 단순함은 대충이 아니다
- 요구사항을 만족하는 최소 해법이어야 한다

### 3. Surgical Changes

뜻:

- 요청 범위만 수정한다
- 주변 코드 개선 욕심을 억제한다
- 자기 변경으로 생긴 미사용 코드만 정리한다
- 기존 스타일을 존중한다

실무 효과:

- 불필요한 파일 수정이 줄어든다
- 리뷰 범위가 좁아진다
- 롤백과 원인 추적이 쉬워진다

주의:

- 주변 맥락까지 읽지 말라는 뜻은 아니다
- 읽는 범위는 넓게, 수정 범위는 좁게가 정확한 해석이다

### 4. Goal-Driven Execution

뜻:

- "어떻게 해"보다 "무엇이 성공인가"를 먼저 정한다
- 검증 가능한 조건을 만든다
- 다단계 작업은 확인 포인트를 둔다

실무 효과:

- 에이전트가 자율적으로 움직이기 쉬워진다
- 결과가 맞는지 판단 기준이 선명해진다

주의:

- 모든 작업을 테스트 코드로 닫으라는 뜻은 아니다
- 문서 작업이면 링크, 구조, 공개 범위, 용어 일관성이 검증 기준이 될 수 있다

## `CLAUDE.md`, `AGENTS.md`, `rules`, `local` 역할 분리

| 위치 | 용도 | 넣어야 할 것 | 넣지 말아야 할 것 |
| --- | --- | --- | --- |
| `CLAUDE.md` | 공통 행동 원칙 | 추측 금지, 과설계 금지, 성공 기준, 협업 방식 | 장문의 절차, 경로별 예외, 개인 취향 |
| `AGENTS.md` | 저장소 운영 설명 | 구조, 폴더 목적, 파일명 규칙, 공개/비공개 기준 | 너무 일반적인 행동 규칙 반복 |
| `.claude/rules/*.md` | 범위 한정 규칙 | 특정 디렉토리, 언어, 테스트, 보안 규칙 | 전 프로젝트 공통 규칙 |
| `CLAUDE.local.md` | 개인 로컬 메모 | 로컬 URL, 개인 선호, 개인 실험 설정 | 팀 공통 규칙 |

실무에서 가장 흔한 문제는 이 역할이 섞이는 것이다.  
역할이 섞이면 `CLAUDE.md`가 길어지고, 규칙 충돌이 늘고, 에이전트는 덜 안정적으로 움직인다.

## 여러 프로젝트에 적용하는 도입 절차

### 1단계. 저장소 사실과 행동 규칙을 분리한다

먼저 다음을 나눈다.

- 저장소 사실: 구조, 명령어, 문서 위치, 운영 원칙
- 행동 규칙: 질문 기준, 변경 범위, 성공 기준, 검증 방식

저장소 사실은 `AGENTS.md`에, 행동 규칙은 `CLAUDE.md`에 둔다.

### 2단계. `CLAUDE.md`는 최소 템플릿부터 시작한다

처음부터 길게 쓰지 않는 편이 낫다.

권장 최소 템플릿:

```md
@AGENTS.md

## 작업 원칙
- 모호하면 추측하지 말고 질문한다.
- 단순 작업은 과잉 질문하지 않는다.
- 요청 범위 밖 기능, 리팩터링, 의존성 추가를 하지 않는다.
- 변경한 모든 줄은 요청과 직접 연결되어야 한다.
- 작업 전 성공 기준을 짧게 정리하고, 작업 후 그 기준으로 검증한다.

## 품질 원칙
- 기존 스타일과 구조를 우선 존중한다.
- 내가 만든 변경으로 인해 생긴 미사용 코드만 정리한다.
- 긴 절차나 특정 경로 규칙은 이 파일에 늘리지 않고 별도 규칙 파일로 분리한다.
```

### 3단계. 필요한 경우에만 `.claude/rules/`를 추가한다

다음은 rules로 분리하는 편이 좋다.

- `docs/`는 한국어로 작성
- `migrations/`는 기존 파일 수정 금지
- `tests/e2e/`는 flaky test 대응 규칙 적용
- `src/billing/`은 계획과 검증 강도 상향

예시:

```md
---
paths:
  - "docs/**/*.md"
---

## 문서 작성 규칙
- 문서는 한국어로 작성한다.
- 공개 불가 정보는 넣지 않는다.
- 기존 번호형 파일 구조를 유지한다.
```

### 4단계. 작업 크기별로 운영 강도를 다르게 한다

#### 작은 작업

예:

- 오타 수정
- 링크 교체
- 한 줄 설정 변경
- 명백한 문장 교정

운영:

- 질문 최소화
- 바로 수정
- 간단 검증 후 종료

#### 중간 작업

예:

- 함수 수정
- API 응답 형식 보정
- 문서 구조 개편
- 테스트 보강

운영:

- 가정 명시
- 성공 기준 2~4개 정의
- 검증 후 종료

#### 고위험 작업

예:

- 인증/권한
- 결제/정산
- 배포 자동화
- 마이그레이션

운영:

- 명시적 계획
- 영향 범위 확인
- 강한 검증
- 필요 시 반론 검토 또는 교차 리뷰

## 프로젝트 유형별 적용 포인트

### 1. 코드 저장소

강조할 것:

- 테스트, 빌드, 린트 기준
- 경로별 책임
- 변경 금지 영역
- 배포 또는 데이터 마이그레이션 규칙

### 2. 문서 저장소

강조할 것:

- 공개 가능 범위
- 문서 카테고리 기준
- 파일명 규칙
- 링크와 용어 일관성

### 3. 혼합 저장소

강조할 것:

- 코드와 문서의 규칙을 분리
- 경로별 rules 적용
- 공개 문서와 내부 문서 기준 분리

## 실패 모드와 보완책

### 실패 모드 1. 질문이 너무 많아진다

원인:

- 모든 모호함을 같은 위험으로 취급

보완:

- 결과를 바꾸는 모호함만 질문
- 작은 작업은 바로 처리
- 질문 횟수 기준을 내부적으로 제한

### 실패 모드 2. `CLAUDE.md`가 길어지고 무거워진다

원인:

- 행동 원칙, 운영 절차, 경로 예외, 개인 취향이 한 파일에 섞임

보완:

- 행동 원칙만 남긴다
- 절차는 별도 문서로 뺀다
- 경로 규칙은 `.claude/rules/`로 보낸다
- 개인 메모는 `CLAUDE.local.md`로 보낸다

### 실패 모드 3. 규칙이 서로 충돌한다

원인:

- 상위 `CLAUDE.md`, 하위 규칙, 로컬 파일이 중복 또는 상충

보완:

- 우선순위를 명시한다
- 중복 규칙을 제거한다
- 새 규칙을 추가할 때 기존 규칙과 겹치는지 먼저 본다

### 실패 모드 4. "외과적 수정"을 과하게 해석한다

원인:

- 주변 맥락을 읽는 것조차 피함

보완:

- 읽는 범위는 넓게, 수정은 좁게 적용
- 관련 없는 개선 욕심만 억제

### 실패 모드 5. `CLAUDE.md`를 만능 해결책처럼 본다

원인:

- 문서만 넣으면 결과가 자동으로 좋아질 거라고 기대

보완:

- 실제 효과는 저장소 구조, 검증 습관, 규칙 품질과 함께 나온다
- 문서는 방향을 주는 도구이지 자동 보정 장치는 아니다

## 팀 적용 체크리스트

도입 전:

- `AGENTS.md`에 저장소 사실이 정리돼 있는가
- 공통 행동 원칙과 프로젝트 사실이 분리돼 있는가
- 공개/비공개 기준이 문서화돼 있는가

도입 시:

- `CLAUDE.md`가 짧고 구체적인가
- 경로별 예외를 `.claude/rules/`로 분리했는가
- 작은 작업과 큰 작업의 운영 강도를 구분했는가

도입 후:

- 요청 외 파일 수정이 줄었는가
- 과설계 리뷰가 줄었는가
- 검증 누락이 줄었는가
- "왜 이것까지 바꿨지?"라는 diff가 줄었는가

## 운영 지표 예시

감각 대신 아래 지표로 보는 편이 낫다.

- 요청 외 파일 수정 비율
- 첫 시도 검증 통과율
- 리뷰에서 지적된 과설계 빈도
- 되돌림 또는 재작업 비율
- 명시적 확인 질문 수

문서 저장소라면 아래도 유효하다.

- 문서 위치 오배치 건수
- 번호 체계 위반 건수
- 공개 불가 정보 유입 건수

## 바로 적용할 때 기억할 한 줄

> `CLAUDE.md`는 "모든 걸 설명하는 파일"이 아니라, 에이전트가 자주 틀리는 행동을 미리 제어하는 짧은 운영 레이어다.

실무 적용의 핵심은 3가지다.

1. `CLAUDE.md`는 짧고 구체적으로 쓴다.
2. 저장소 사실 정보는 `AGENTS.md`에 두고 import 한다.
3. 작업 크기와 위험도에 따라 규칙 강도를 조절한다.

이 3가지만 지켜도 diff는 작아지고, 과설계는 줄고, 에이전트의 엇나간 창의성도 눈에 띄게 줄어든다.
