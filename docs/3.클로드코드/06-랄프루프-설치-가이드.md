# 랄프 루프(Ralph Loop) 설치 가이드

Claude Code에서 자율형 AI 코딩을 위한 **랄프 루프** 설치 및 운용 가이드입니다.

---

## 1. 핵심 개념

```mermaid
flowchart LR
    subgraph traditional [기존 AI 코딩 - 문제점]
        T1[긴 대화 누적]
        T2[컨텍스트 포화]
        T3[추론 능력 퇴화]
        T4[환각 현상]
        T1 --> T2 --> T3 --> T4
    end
    
    subgraph ralph [랄프 루프 - 해결책]
        R1[새 AI 인스턴스]
        R2[PRD/Progress 읽기]
        R3[작업 실행]
        R4[Git 커밋]
        R1 --> R2 --> R3 --> R4
        R4 -->|세션 종료 후 재시작| R1
    end
    
    traditional -.->|패러다임 전환| ralph
```

| 구분 | 기존 방식 | 랄프 루프 |
|------|----------|----------|
| **컨텍스트** | 누적되어 오염 | 매번 새로 시작 |
| **메모리** | 대화 이력 의존 | 파일 시스템 + Git |
| **안정성** | 장시간 작업 시 퇴화 | 일관된 품질 유지 |

> **핵심 철학**: "LLM은 자신의 과제를 스스로 채점하는 능력이 부족하다" - 외부 검증 루프 필수

## 2. Persistence 매커니즘

```mermaid
flowchart TB
    subgraph memory [외부화된 상태 저장소]
        PRD[prd.json<br/>원자적 작업 목록]
        PROGRESS[progress.txt<br/>시행착오 기록]
        GIT[Git History<br/>코드 변경 이력]
    end
    
    subgraph loop [새 AI 세션]
        READ[상태 읽기]
        WORK[작업 실행]
        WRITE[상태 갱신]
    end
    
    PRD --> READ
    PROGRESS --> READ
    GIT --> READ
    READ --> WORK --> WRITE
    WRITE --> PRD
    WRITE --> PROGRESS
    WRITE --> GIT
```

| 컴포넌트 | 파일 형식 | 역할 |
|---------|----------|------|
| **PRD** | `prd.json` | 원자적 작업 목록 및 완료 상태 관리 |
| **Progress Log** | `progress.txt` | 이전 루프의 패턴, 시행착오, 특이사항 기록 |
| **Git History** | Git Repo | 실제 코드 변경 사항 및 커밋 메시지 보존 |

## 3. 설치 방법: Anthropic 공식 플러그인

```mermaid
flowchart LR
    subgraph install [설치 단계]
        I1[플러그인 설치]
        I2[설치 확인]
        I3[루프 시작]
    end
    
    subgraph mechanism [동작 방식]
        M1[세션 내 Stop Hook]
        M2[종료 시도 가로채기]
        M3[루프 자동 재개]
    end
    
    I1 --> I2 --> I3
    I3 --> M1 --> M2 --> M3
```

**설치 및 사용:**

```bash
# Claude Code 터미널에서 실행

# 1. 플러그인 설치
/plugin install ralph-wiggum@claude-plugins-official

# 2. 설치 확인 (Tab 자동완성)
/ralph [Tab]

# 3. 루프 시작
/ralph-loop "작업 내용" --completion-promise "완료 신호" --max-iterations 20
```

**주요 옵션:**

| 옵션 | 설명 | 예시 |
|-----|------|------|
| `--completion-promise` | 완료 조건 신호 | `"모든 테스트 통과"` |
| `--max-iterations` | 최대 반복 횟수 | `20` |

> **참고**: 단일 컨텍스트 윈도우 내에서 동작하므로 단기/명확한 작업에 적합

## 4. 전체 워크플로우

```mermaid
flowchart TB
    subgraph setup [1. 초기 설정]
        S1[PRD 작성]
        S2[prd.json 생성]
        S3[progress.txt 초기화]
    end
    
    subgraph loop [2. 자율 루프]
        L1[새 AI 세션 시작]
        L2[prd.json에서 다음 작업 식별]
        L3[progress.txt로 이전 실패 참조]
        L4[코드 작성 및 테스트]
        L5[Git 커밋]
        L6{모든 작업 완료?}
        
        L1 --> L2 --> L3 --> L4 --> L5 --> L6
        L6 -->|아니오| L1
    end
    
    subgraph verify [3. 검증]
        V1[테스트 통과 확인]
        V2[린트/빌드 검증]
    end
    
    setup --> loop
    L6 -->|예| verify
```

## 5. 트러블슈팅

### 컨텍스트 오염 증상

```mermaid
flowchart LR
    subgraph symptoms [증상]
        SY1[동일 해결책 반복 실패]
        SY2[완료된 작업 인식 못함]
    end
    
    subgraph solutions [해결책]
        SO1["clear 또는 compact 명령"]
        SO2["no-continue 옵션"]
        SO3["progress.txt에 금지 패턴 기록"]
    end
    
    symptoms --> solutions
```

| 해결책 | 명령어/방법 | 설명 |
|-------|-----------|------|
| **수동 초기화** | `/clear` 또는 `/compact` | 컨텍스트 초기화/요약 |
| **강제 신규 세션** | `--no-continue` | 이전 세션 ID 공유 안 함 |
| **명시적 가이드** | progress.txt 수정 | "방법 X는 Y 오류 발생, 사용 금지" 기록 |

## 6. 비용 관리

| 전략 | 설정 | 효과 |
|-----|------|------|
| **반복 횟수 제한** | `max-iterations: 10~20` | 초기 테스트 시 토큰 절약 |
| **모델 혼합** | 단순 작업: Haiku, 복잡 작업: Sonnet | 비용 대비 효율 최적화 |

## 7. 보안 권장사항

```mermaid
flowchart TB
    subgraph security [보안 아키텍처]
        D[Docker Sandbox]
        P[권한 관리]
        M[최소 권한 원칙]
    end
    
    subgraph commands [명령어]
        C1[docker sandbox run claude]
        C2[settings.json permissions]
        C3[API 키 최소화]
    end
    
    D --> C1
    P --> C2
    M --> C3
```

> **필수**: `--dangerously-skip-permissions` 사용 시 반드시 Docker 샌드박스 내에서 실행
